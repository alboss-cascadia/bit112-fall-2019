Make sure file extensions are visible

Unless you’ve forced it to behave, your operating system is lying to you. It's not showing you the complete filenames. This will drive you crazy in a web or programming class.

By default, both Windows and MacOS are set up to hide the extension, the dot and the letters at the end of the file name.

If you haven't already done this, change the default on your computer so you can see complete filenames including the “extension,” the dot and the letters at the end of the file name.

  • Microsoft Windows
  • Mac OS
  • Linux assumes you are competent and doesn't hide the file extensions from you by default. Linux wins this round.

Cache

Bowsers store a temporary copy, or cache, of pages you've visited recently. They're supposed to replace the copy when the page changes, but it doesn't always work. So, any time you're not seeing a change you know you made, it might be your web browser playing tricks on you. Here are instructions for most browsers:

Console error messages

All the browsers handle errors differently. Most of them will highlight where they think the problem is, but it's still algorithms and artificial intelligence and sometimes they get it wrong.

For example:

<h1>Heading

<p>Blah blah blah</p>

Will usually return an error saying the <p> is misplaced because it's inside a heading. The actual error is that I forgot the closing </h1>.

<p>This <a href="page.html">link will be a problem.</p></a>

The error message will probably say something like there's a closing </a> that doesn't match anything. The problem is that my nesting is off; I needed to </a> before I </p>.

So, even when it says there's an error, sometimes the clue it gives you isn't quite right.

You can open a page from your computer in a web browser and use the inspector on that. We aren't doing anything that will only work on a server. Any of the browsers can open a file from your computer, and their inspector consoles will work.

Of course, if you would like someone else in the class (including me) to help figure something out, the code needs to be in GitHub unless one of us is sitting at your computer with you. And, even once it's working on your computer, it needs a final check on the server just to make sure it works there as well. Sometimes things work on your computer but not on the server. It doesn't happen often, but when it does happen it's inconvenient.

How does the em or rem work?

An em is the width of the widest character in the charset, which in our alphabet is the M.

If you don't specify any absolute values (px, pt, in, etc.) then it will work off the defaults that are set in the browser. This is awesome for accessibility because it means if I have set my base font size to be 24px instead of 14px, the headings will resize in proportion to my default, rather than whatever arbitrary size some designer picked that's always way too small for me to see.

This is terrible for designers because they don't like giving that much control to users. They set the design to work within certain constraints, and if people override those constraints you get unexpected results, like text that is bigger than the box that is supposed to contain it, or things that don't print out right, or buttons that don't line up with their labels.

Usability/accessibility people and designers should never date. It doesn't end well.

To make the math easy in my example here, I'll say I've set the default to be 10px.

html { font-size: 10px }

Therefore 1em is 10px.

Here's where it gets tricky. With rem, I set the default for the html element to 10px and everything will be relative to that. The rem is always based on the root element of the document, which is the html element. So if I say

h1 { font-size: 3.6rem }

that's 3.6 x 10px, or 36px. Rem is super-easy for figuring out font-size, and if you change the font-size of the the root element, you just changed it for the whole document because it's all multiplying off that.

The em is slightly different for one practical reason.

Em inherits from its parent element. It will work exactly like rem until you override something. Let's say I set

html { font-size: 10px }
h1 { font-size: 3.6em }

but then I set

aside { font-size: 20px }

My html will turn out kind of like this:

<html>
<body>
<h1>36px</h1>
<p>10px</p>
<aside>
<h1>72px because it's 3.6x the parent element, which is aside, which is 20px</h1>
<p>20px</p>
</aside>
<p>Back to 10px</p>
</body>
</html>

There's a fairly understandable article on this admittedly abstract concept, at https://j.eremy.net/confused-about-rem-and-em/. I'm not sure I agree with his conclusion, but his explanations are good, and the part called Save Lives with EM is a great example of why we use em and rem instead of px.
